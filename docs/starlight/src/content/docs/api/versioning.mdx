# Versioning & Compatibility

ProtocolSoup follows a clear versioning strategy to keep integrations stable while allowing the platform to evolve.

## Version Scheme

All API endpoints are versioned under a **`/v1`** prefix (implicit in the gateway routing). The version number increments only on breaking changes.

| Change type | Version impact | Example |
|-------------|---------------|---------|
| New endpoint added | None (additive) | `POST /v1/oid4vci/credential` added |
| New optional field in response | None (additive) | `"metadata"` field added to token response |
| Field renamed or removed | Major bump | `"client_id"` → `"clientId"` |
| Endpoint path changed | Major bump | `/oauth/token` → `/oauth2/token` |

## Compatibility Guarantees

Within a major version:

- **Existing endpoints** will not be removed or have their paths changed.
- **Existing response fields** will not be renamed or removed.
- **New optional fields** may be added to JSON responses at any time -- clients should ignore unknown fields.
- **New endpoints** may be added at any time.
- **Error response shapes** (`{"error": "...", "error_description": "..."}`) remain stable.

## OpenAPI Specifications

Each service publishes a versioned OpenAPI 3.1 specification:

| Service | Spec path | Description |
|---------|-----------|-------------|
| Gateway | `/openapi/gateway.yaml` | Health, discovery, sessions, Looking Glass |
| Federation | `/openapi/federation.yaml` | OAuth 2.0, OIDC, SAML, OID4VCI, OID4VP |
| SCIM | `/openapi/scim.yaml` | User/group provisioning, bulk operations |
| VC | `/openapi/vc.yaml` | Credential issuance, presentations, verification |

Specs are the source of truth for request/response contracts. Browse them interactively in the [API Reference](/api/reference/).

## Deprecation Policy

When a breaking change is planned:

1. The existing behavior is marked **deprecated** in the OpenAPI spec and release notes.
2. A minimum **90-day** deprecation window is provided before removal.
3. The replacement endpoint or field is documented alongside the deprecation notice.

## Upgrade Planning

- Pin your integration to a specific container image tag (e.g., `ghcr.io/parlesec/protocolsoup-gateway:v1.2.0`) rather than `latest`.
- Review the [release and tag policy](/deploy/release-and-tag-policy/) for the image tagging convention.
- Subscribe to the GitHub repository's releases for change notifications.
